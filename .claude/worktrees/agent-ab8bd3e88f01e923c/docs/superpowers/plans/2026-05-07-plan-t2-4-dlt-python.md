## Design Decisions

1. **Why dlt (not Airbyte / Fivetran / Singer / custom).** dlt is Python-native, declarative, and lightweight enough to run inside an E2B sandbox without an external orchestrator. Airbyte requires Docker-in-Docker + a meta-service mesh; Fivetran is SaaS-only; Singer requires writing taps + targets in two languages. dlt is a single Python lib with `@dlt.resource` / `@dlt.source` decorators that match the canvas affordance perfectly: a developer writing a pipeline in Atlas writes Python functions that yield records, and dlt handles schema inference + DuckDB load. Maps cleanly to a "give me a daily ETL job that loads CSV → DuckDB" prompt.

2. **Why DuckDB as the default destination (not Postgres / Snowflake / BigQuery).** DuckDB runs embedded — no external service, no credentials, no network. A `.duckdb` file in `/code/atlas.duckdb` is the entire warehouse. Diego/priya can query it from `/runs` (via the FastAPI status app) or from a dbt model directly; ama gets the abstraction "your data lives here" without needing to provision anything. The user can always swap dlt's destination to Postgres/Snowflake later via a one-line config — DuckDB is just the zero-friction local default. Matches the OSS-stack pivot (ADR-001).

3. **Why dbt-core + dbt-duckdb (not just raw SQL or sqlmesh).** dbt is the de-facto T-side of ELT — it's what users coming from Modern Data Stack experience expect. dbt-duckdb makes dbt usable without a remote warehouse. Shipping a `dbt_project/` skeleton makes "now write a transform that joins these two sources" a one-file edit (just add `models/foo.sql`) rather than a multi-file infrastructure change. sqlmesh is technically more powerful but has thinner reference material in researcher catalog; dbt's docs + community examples are richer.

4. **Why pandas + pyarrow as transitive deps.** dlt uses pyarrow for schema introspection + columnar transfer; pandas is the lingua franca for ad-hoc data work that diego/priya will reach for inside dlt resources. Pre-installing both means the developer's diff doesn't need to also write pyproject.toml patches when the user asks for "a quick groupby on the loaded data". (Same justification as why atlas-fastapi pre-installs sqlalchemy + alembic.)

5. **Why bundle a FastAPI status app on port 3000.** Data pipelines don't have a UI. The E2B preview iframe needs *something* served on port 3000 — otherwise diego/priya see a blank/broken iframe and conclude the pipeline failed. The status app reads the DuckDB file and exposes `/health` (liveness), `/runs` (last N pipeline-run records via `dlt.pipeline.runs`), and `/lineage` (DAG visualization showing source → resource → table → dbt-model). For ama persona the same endpoints render a simplified "Last run: 2m ago, 1,200 rows loaded, status: healthy" status page. This is the load-bearing decision that makes the data-pipeline preview meaningful in the Atlas canvas.

6. **Why also expose `/lineage` (not just `/health` + `/runs`).** Lineage is *the* killer demo for data-pipeline projects. It's how diego/priya validate the pipeline matches the brief — and how ama persona understands "where does my data come from". Reading lineage from dlt's metadata + dbt's manifest is a 30-line endpoint, far cheaper than building a separate visualization tool. The endpoint returns a JSON DAG that the frontend (or curl) can render.

7. **Why a single canonical example pipeline (`example_pipeline.py`).** Same justification as atlas-fastapi's `app/main.py` smoke test. The example pipeline (1) proves the dlt+DuckDB stack works end-to-end at template-build time; (2) gives the developer role a structural reference to mimic when writing the user's diff; (3) populates the DuckDB file with at least one row so the status app's `/runs` endpoint isn't empty. Pattern: extract from a fake static source (a hardcoded list of records), load to a `example_data` table.

8. **Why dbt's profiles.yml co-located with dbt_project.yml (not in $HOME/.dbt/).** Sandbox isolation. Default dbt looks for profiles in `$HOME/.dbt/profiles.yml` — but in the sandbox, `$HOME` is volatile + the developer's diff can't touch outside `/code`. Shipping the profile alongside the project (with a `profiles_dir` flag in `dbt_project.yml`) keeps everything inside `/code` so dbt commands work without env-var manipulation.

9. **Why `pyproject.toml` + `uv.lock` (not requirements.txt).** Same as Plan T.1 reasoning: reproducible builds + `uv` is faster + matches the existing `pnpm py:gen` toolchain.

10. **Why route `data-pipeline` to `atlas-dlt-python` (not leave it in fallback to `atlas-next-ts-v2`).** Plan T.1 left `data-pipeline` in the fallback bucket explicitly to defer until a real Python data-stack template exists. T.2.4 *is* that real template — so flipping the case is the entire point. Per-project override via `ATLAS_DEFAULT_SANDBOX_TEMPLATE` still wins for users who want to pin a different stack.

11. **Why ship the v1 template-id as `<TBD-after-build>` placeholder.** Same pattern as Plan T.1 — implementing engineer commits the e2b.toml without a real template_id; operator post-merge runs `build-template.sh`, captures the printed ID, commits a one-line follow-up updating `template_id`. Avoids embedding a fake ID that will silently break.

12. **Why the existing Plan T.1 prompt-registry plumbing requires no changes.** Plan T.1 designed `getSandboxContextPrompt()` to take any `templateName` string and return the matching fragment with graceful fallback. T.2.4 just adds one more entry to the `REGISTRY` record. No types change, no consumer code changes, no new exports.

13. **Why a Researcher skill named `assemble-brief-data-pipeline.md` (mirroring T.1's pattern).** Plan T.1 established the `assemble-brief-${artifactKind}.md` naming convention. The data-pipeline brief cites Fivetran's connector catalog (for source-of-truth on extraction patterns), Airbyte's open-source connector library (alternative reference for users who balk at SaaS), dlt's official examples (for canonical pipeline shape), dbt's docs (for transformation idioms — sources, refs, tests), and "Modern Data Stack" community patterns (for orchestration choices: cron vs Airflow vs Dagster vs Prefect).

```python
"""Canonical dlt example pipeline.

Demonstrates the @dlt.resource / @dlt.source / pipeline.run() pattern that
the developer role mimics when writing the user's data-pipeline diff.

Loads a small static record set into a DuckDB destination at
./atlas.duckdb (or the override passed to run_pipeline).
"""

from __future__ import annotations

from typing import Iterator

import dlt
from dlt.common.pipeline import LoadInfo


@dlt.resource(name="example_data", write_disposition="replace")
def example_source() -> Iterator[dict]:
    """A trivial dlt resource that yields hard-coded records.

    Replace this with a real source (HTTP API, SQL DB, S3 CSV, ...) by
    replacing the body — the @dlt.resource contract is just "yield dicts".
    """
    yield {"id": 1, "value": "alpha", "loaded_at": "2026-05-07T00:00:00Z"}
    yield {"id": 2, "value": "beta", "loaded_at": "2026-05-07T00:00:00Z"}
    yield {"id": 3, "value": "gamma", "loaded_at": "2026-05-07T00:00:00Z"}


def run_pipeline(db_path: str = "./atlas.duckdb") -> LoadInfo:
    """Run the example pipeline → DuckDB.

    Args:
        db_path: Path to the DuckDB file. Defaults to ./atlas.duckdb so the
                 status app finds it relative to the sandbox cwd.

    Returns:
        LoadInfo from dlt — useful for the status app's /runs endpoint.
    """
    pipeline = dlt.pipeline(
        pipeline_name="atlas_example",
        destination=dlt.destinations.duckdb(db_path),
        dataset_name="atlas_dataset",
    )
    return pipeline.run(example_source())


if __name__ == "__main__":
    info = run_pipeline()
    print(info)
```

```python
"""FastAPI status app served on port 3000.

Data pipelines have no UI. This tiny status surface makes the E2B preview
iframe show SOMETHING — pipeline runs + lineage. Read-only against the
DuckDB file at ./atlas.duckdb.
"""

from __future__ import annotations

import os
from pathlib import Path
from typing import Any

import duckdb
from fastapi import FastAPI

DB_PATH = os.getenv("ATLAS_DUCKDB_PATH", "./atlas.duckdb")

app = FastAPI(
    title="Atlas Data Pipeline",
    version="0.1.0",
    description="Status surface for an Atlas-generated data pipeline (dlt + DuckDB + dbt)",
)


@app.get("/")
def root() -> dict[str, str]:
    return {"name": "Atlas Data Pipeline", "version": "0.1.0", "stack": "dlt-python"}


@app.get("/health")
def health() -> dict[str, str]:
    """Liveness probe — does NOT require the DuckDB file to exist."""
    return {"status": "ok", "stack": "dlt-python", "atlas": "sandbox-ready"}


@app.get("/runs")
def runs(limit: int = 20) -> dict[str, Any]:
    """List the last N pipeline runs from DuckDB's _dlt_loads table.

    Returns an empty list if the DuckDB file does not exist yet (pipeline
    hasn't run).
    """
    if not Path(DB_PATH).exists():
        return {"runs": [], "db_path": DB_PATH, "note": "DuckDB file not yet created — run a pipeline first."}
    try:
        with duckdb.connect(DB_PATH, read_only=True) as conn:
            rows = conn.execute(
                """
                SELECT load_id, schema_name, status, inserted_at
                FROM atlas_dataset._dlt_loads
                ORDER BY inserted_at DESC
                LIMIT ?
                """,
                [limit],
            ).fetchall()
            return {
                "runs": [
                    {"load_id": r[0], "schema": r[1], "status": r[2], "inserted_at": str(r[3])}
                    for r in rows
                ],
                "db_path": DB_PATH,
            }
    except duckdb.CatalogException:
        # _dlt_loads table not yet created (no pipeline has run successfully).
        return {"runs": [], "db_path": DB_PATH, "note": "_dlt_loads table not yet created."}


@app.get("/lineage")
def lineage() -> dict[str, Any]:
    """Return a DAG of source → resource → table → dbt-model.

    v1: discovers nodes/edges from DuckDB schema introspection. dbt nodes
    are added when dbt_project/target/manifest.json exists (after a dbt
    parse run); otherwise only dlt-side nodes are returned.
    """
    nodes: list[dict[str, str]] = []
    edges: list[dict[str, str]] = []

    if Path(DB_PATH).exists():
        try:
            with duckdb.connect(DB_PATH, read_only=True) as conn:
                tables = conn.execute(
                    """
                    SELECT table_schema, table_name
                    FROM information_schema.tables
                    WHERE table_schema NOT IN ('information_schema', 'pg_catalog', 'main')
                      AND table_name NOT LIKE '\\_dlt%' ESCAPE '\\'
                    """
                ).fetchall()
                for schema, table in tables:
                    node_id = f"{schema}.{table}"
                    nodes.append({"id": node_id, "kind": "table", "label": table})
                    edges.append({"from": f"source:{schema}", "to": node_id})
                    if not any(n["id"] == f"source:{schema}" for n in nodes):
                        nodes.append({"id": f"source:{schema}", "kind": "source", "label": schema})
        except duckdb.Error:
            pass

    manifest = Path("dbt_project/target/manifest.json")
    if manifest.exists():
        import json as _json

        try:
            data = _json.loads(manifest.read_text())
            for unique_id, model in (data.get("nodes") or {}).items():
                if model.get("resource_type") == "model":
                    label = model.get("name", unique_id)
                    nodes.append({"id": unique_id, "kind": "dbt_model", "label": label})
                    for dep in model.get("depends_on", {}).get("nodes", []):
                        edges.append({"from": dep, "to": unique_id})
        except (ValueError, KeyError):
            pass

    return {"nodes": nodes, "edges": edges, "db_path": DB_PATH}
```

# atlas-dlt-python E2B template

Python 3.12 + dlt 1.4+ + DuckDB 1.x + dbt-core 1.9+ + dbt-duckdb 1.9+ + pandas 2.x + pyarrow 18+ + FastAPI 0.115 + pytest + ruff via uv.

## Why a FastAPI app in a data-pipeline template?

Data pipelines have no UI. The E2B preview iframe needs *something* served on port 3000 so users see signal that the pipeline is alive. This template bundles a tiny FastAPI status app that exposes:

- `GET /` — app metadata (name, version, stack)
- `GET /health` — `{"status": "ok", "stack": "dlt-python", "atlas": "sandbox-ready"}`
- `GET /runs?limit=20` — last N pipeline runs (read from DuckDB's `_dlt_loads` table)
- `GET /lineage` — DAG of source → resource → table → dbt-model (read from DuckDB schema + `dbt_project/target/manifest.json` if present)

The status app is read-only against the DuckDB file at `./atlas.duckdb`. The dlt pipelines write to that same file. dbt models read from it. Everything lives in a single `.duckdb` file inside `/code`.

# Assemble Brief — Data Pipelines

Use this fragment when `designIntent.artifactKind === "data-pipeline"`. References are data-platform tools and Modern Data Stack patterns — NOT landing pages or generic backend APIs.

## Reference catalog (cite at least one per category)

### Connector / extraction patterns
- **Fivetran** (https://www.fivetran.com/connectors) — closed-source SaaS, but its connector catalog is the canonical reference for "what does a 'load Stripe data' pipeline look like". Cite the *shape* of the connector (incremental keys, retry semantics, schema evolution) — not the SaaS itself.
- **Airbyte** (https://docs.airbyte.com/integrations/) — open-source, runs anywhere. Connector specs are public; copy their patterns for source authentication + sync modes (full_refresh vs incremental_append vs incremental_dedup).
- **Singer** (https://www.singer.io) — older OSS spec. Reference if the user asks for a tap+target architecture specifically.
- **dlt** (https://dlthub.com/docs/) — Python-native, declarative. The `@dlt.resource` / `@dlt.source` pattern is what `atlas-dlt-python` ships. Cite the dlt verified-source examples (https://dlthub.com/docs/dlt-ecosystem/verified-sources) for source-shape — they're the closest match to what Atlas's developer role will write.

### Transformation patterns
- **dbt** (https://docs.getdbt.com) — de-facto standard for SQL transformations. Cite dbt's sources/refs/tests/snapshots/exposures concepts. Materialization choice (view vs table vs incremental) is load-bearing for the brief.
- **dbt project structure best practices** (https://docs.getdbt.com/best-practices/how-we-structure/1-guide-overview) — staging / intermediate / marts layering. Cite this when the user asks for "a real dbt project, not a toy".
- **sqlmesh** (https://sqlmesh.readthedocs.io) — newer alternative; reference only if the user explicitly mentions blue/green deploys or contracts.

### Destinations / warehouses
- **DuckDB** (https://duckdb.org/docs) — embedded analytical DB. Default destination for atlas-dlt-python. Cite for "I want a local-first warehouse with no creds" cases.
- **Snowflake / BigQuery / Postgres** — cite for the destination-swap path: dlt's `destination=` is one config line. Don't recommend Snowflake/BigQuery as default for Atlas projects (cred friction); offer them as the v2 path.

### Orchestration patterns
- **cron + a single Python script** — simplest. Cite for "I just want a daily job" briefs.
- **Airflow** (https://airflow.apache.org) — reference for "I have multiple pipelines with dependencies between them".
- **Dagster** (https://docs.dagster.io) — newer, asset-graph-first. Reference for "I want lineage built in".
- **Prefect** (https://docs.prefect.io) — Python-flow first. Reference for "I want decorators + retries without YAML".

### Modern Data Stack patterns
- **a16z's "Emerging Architectures for Modern Data Infrastructure"** — taxonomy reference: ingestion → storage → transformation → analytics → activation.
- **Locally Optimistic / dbt Labs / Airbyte / Fivetran blogs** — pattern catalog: "the small data warehouse", "ELT vs ETL", "incremental models", "data contracts".

## Quality bar

- The brief names a **specific source** (Stripe, Postgres, S3 CSV, an HTTP API) — not "data".
- The brief names a **specific destination** (DuckDB by default; Snowflake/Postgres if the user pushes back).
- The brief names a **frequency** (one-shot, hourly, daily, on-demand). Default = daily for "background ETL" requests.
- The brief specifies **incremental vs full-refresh** semantics and the **incremental key** (timestamp column, sequence ID, etc.). Default = full-refresh for v1; incremental only when the source is large.
- The brief names at least **one transformation** — even if it's a passthrough view + one derived column. dbt without models is just storage.
- For multi-tenancy / PII briefs: explicit isolation strategy (tenant column + RLS, OR separate schemas, OR separate DuckDB files). Don't hand-wave.

## Anti-patterns

- Do NOT propose Spark / Flink / Beam clusters for atlas-dlt-python projects. The template is single-process Python; pretending otherwise is a lie.
- Do NOT recommend writing custom orchestration code in v1 — cron / `python -m pipelines.foo` is enough until the user explicitly asks for a DAG.
- Do NOT recommend Kafka / event-streaming sources without a clear case. dlt + cron handles 90% of "load X daily" briefs.
- Do NOT cite landing pages, generic SaaS marketing sites, or frontend tools. Stay in the data domain.
- Do NOT propose dbt without first declaring sources in `schema.yml`. "Just write a SQL file" without a source declaration is the #1 dbt onboarding pitfall.

| FastAPI status app fails to start (e.g., uvicorn not in PATH) → preview iframe blank. | Low | `ready_cmd = "curl -fsS http://localhost:3000/health"` in e2b.toml; E2B retries until /health responds before declaring the sandbox ready. |

| pyarrow + pandas wheels add ~150MB to the image. | Low | Acceptable cost — these are pre-installed because dlt depends on them transitively + pandas is a baseline expectation for diego/priya. |
| dbt's `target/manifest.json` not generated until `dbt parse` or `dbt compile` runs → /lineage endpoint shows only dlt nodes, no dbt nodes. | Medium | Document in README: "Run `cd dbt_project && uv run dbt parse --profiles-dir .` to populate manifest". Status app degrades gracefully (returns dlt-only DAG). |
| Operator forgets to republish template after merge → live atlas-web fails sandbox creation for data-pipeline projects. | High | PR description explicitly calls out the operator step. Same pattern as Plan T.1; team has the muscle memory. |
