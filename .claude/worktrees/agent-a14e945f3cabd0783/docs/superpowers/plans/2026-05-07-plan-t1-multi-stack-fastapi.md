> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

## Design Decisions

1. **Why `uv` over `pip`.** `uv` is ~10× faster on cold installs, increasingly the Python-community standard, and matches the existing `pnpm py:gen` toolchain in the monorepo. Single `uv sync` at Dockerfile build time replaces the multi-step pip + virtualenv dance.

2. **Why `pyproject.toml` + `uv.lock` (not requirements.txt).** Reproducible builds. `uv.lock` pins all transitive deps cryptographically. `pyproject.toml` declares the app metadata + ruff/pytest config in one place.

3. **Why `/health` and `/docs` as smoke endpoints.** `/health` is the canonical Kubernetes readiness check pattern — every backend ships one; verifying it works proves the sandbox boot is correct. `/docs` is FastAPI's auto-generated Swagger UI — shows up the moment FastAPI is up; gives diego/priya users an immediate "this is real" moment without writing extra code.

4. **Why per-template prompt fragments (not one mega-prompt with switches).** Token cost. A switch-based prompt would force the LLM to read all stacks' info every call. Per-template fragments mean Sonnet only sees the prompt for the actual target stack — keeps the developer prompt tight + relevant.

5. **Why route via `canvasManifest.artifactKind` (not via a separate architect output field).** `canvasManifest` already carries this for the canvas shell (Plan S.4 ships it). Reusing the same field keeps the data model simple — one source of truth for "what is this project."

6. **Why behind `ATLAS_FF_MULTI_STACK` (not always-on).** Behavioural lock. Flag-OFF preserves today's exact tree — every project still provisions `atlas-next-ts-v2`. Flag-ON enables per-artifact-kind routing. Standard pattern matches Plan S series.

7. **Why fallback to `atlas-next-ts-v2` for unsupported kinds (not error).** Graceful degrade. v1 only ships `atlas-fastapi`; mobile-app/data-pipeline/cli-tool prompts fall through to Next.js (works for some, not all). v2 sub-plans replace the fallback per kind. Better to give the user something running than block on a missing template.

8. **Why per-template README.** Each template's quirks (boot command, dep install, smoke-test URL) differ. Co-locating docs with the template is the easiest pattern for operators.

9. **Why ship the v1 template-id as `<TBD-after-build>` placeholder.** Same pattern as Plan S.1 — implementing engineer commits the e2b.toml without a real template_id; operator post-merge runs `build-template.sh`, captures the printed ID, commits a one-line follow-up updating `template_id`. Avoids embedding a fake ID that will silently break.

RUN apt-get update \
 && apt-get install -y --no-install-recommends python3.12 python3.12-venv python3-pip curl ca-certificates \
 && rm -rf /var/lib/apt/lists/* \
 && curl -LsSf https://astral.sh/uv/install.sh | sh \
 && cp /root/.local/bin/uv /usr/local/bin/uv

WORKDIR /code

# Hand-author the app structure so the build is deterministic.
RUN mkdir -p app tests

COPY pyproject.toml ./
COPY app/__init__.py app/main.py ./app/
COPY tests/__init__.py tests/test_main.py ./tests/

# Install via uv (creates .venv inside /code).
RUN uv sync --frozen 2>/dev/null || uv sync \
 && chown -R user:user /code

USER user

EXPOSE 3000

# Note: e2b.toml's start_cmd overrides this CMD at sandbox boot.
CMD ["sh", "-c", "uv run uvicorn app.main:app --host 0.0.0.0 --port 3000"]
```

# Build + push the atlas-fastapi E2B template image.
# Usage: ./scripts/build-template.sh
# Requires: E2B_API_KEY env var set; npx + node 22+ available.

import type { ArtifactKind } from "@atlas/canvas-runtime";

const DEFAULT_TEMPLATE = "atlas-next-ts-v2";

/**
 * Map architect's `canvasManifest.artifactKind` to an E2B template name.
 *
 * v1: ships atlas-next-ts-v2 (frontend) + atlas-fastapi (backend-rest-api).
 * Other kinds fall back to atlas-next-ts-v2 as a safe default; v2 sub-plans
 * (T.2.1, T.2.2, ...) replace the fallbacks with dedicated templates.
 *
 * When multiStackFlagOn=false, returns DEFAULT_TEMPLATE regardless of kind
 * — preserves today's exact behavior under flag-OFF.
 */
export function templateForArtifactKind(
  kind: ArtifactKind | undefined,
  opts: { multiStackFlagOn: boolean }
): string {
  if (!opts.multiStackFlagOn) return DEFAULT_TEMPLATE;
  if (!kind) return DEFAULT_TEMPLATE;
  switch (kind) {
    case "frontend-app":
      return "atlas-next-ts-v2";
    case "backend-rest-api":
      return "atlas-fastapi";
    case "backend-graphql":
    case "data-pipeline":
    case "mobile-app":
    case "cli-tool":
      // v2 sub-plans (T.2.x) replace these fallbacks.
      return DEFAULT_TEMPLATE;
  }
}
```

/**

 *
 * Precedence (highest first):
