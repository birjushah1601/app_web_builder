# Plan T.2.3 — atlas-expo-rn (Expo SDK 52 + RN + NativeWind + Expo Router) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Add an Expo + React Native E2B template (`atlas-expo-rn`) for the `mobile-app` artifact kind. Wire it into the per-artifact-kind template router introduced in Plan T.1, register a per-template developer prompt fragment, ship a per-artifact-kind Researcher skill, and update the `mobile-app` routing case so `templateForArtifactKind("mobile-app", { multiStackFlagOn: true }) === "atlas-expo-rn"` (replacing the v1 fallback to `atlas-next-ts-v2`). Behind the existing `ATLAS_FF_MULTI_STACK=false` flag — flag-OFF preserves today's exact pipeline.

**Architecture:** Three coordinated changes mirroring T.1's shape. **(1)** New `packages/sandbox-e2b/templates/atlas-expo-rn/` carrying an Expo SDK 52 + React Native 0.76+ + expo-router 4.x + NativeWind 4.x + TypeScript 5.6 (strict) project. The dev server runs `pnpm expo start --web --port 3000` so the E2B preview iframe renders the **react-native-web** build of the app — native iOS/Android simulator support is explicitly v2 work. The template ships an `app/_layout.tsx` (Expo Router root) + `app/index.tsx` (NativeWind smoke screen) + `app/(tabs)/_layout.tsx` (tabs example) so the developer LLM has a real shape to extend. Built + pushed to E2B by an operator post-merge (mirrors T.1's deferred republish). **(2)** `packages/role-developer/src/sandbox-context-prompts/expo-rn.ts` — positive-list prompt fragment registered in `sandbox-context-registry.ts` under the `atlas-expo-rn` key. The prompt teaches the developer LLM about Expo Router file conventions (`app/_layout.tsx`, `app/index.tsx`, `app/(tabs)/_layout.tsx`), NativeWind `className` syntax (Tailwind classes via the babel plugin), RN component idioms (`<View>` not `<div>`, `<Text>` not raw strings), the Link component from `expo-router` (not HTML `<a>`), no `document`/`window` assumptions, and the web-only preview limitation. **(3)** `apps/atlas-web/lib/sandbox/template-router.ts` updated so the `mobile-app` artifact kind routes to `atlas-expo-rn` when the multi-stack flag is on (T.1 left this as a fallback to `atlas-next-ts-v2`).

**Tech Stack:** TypeScript 5.6 (strict) · Node 22 · pnpm 9 · Expo SDK 52 · React Native 0.76+ · expo-router 4.x · NativeWind 4.x · Tailwind 3.4 · jest + jest-expo · E2B SDK + CLI · Docker.

**Prerequisites:**
- Plan T.1 merged to `main` and the `atlas-fastapi` template republished to E2B (so the per-artifact-kind routing primitive is live + tested).
- Node 22 + pnpm 9.
- Docker (for local Dockerfile sanity-test of the Expo web build).
- `E2B_API_KEY` available — needed for the operator post-merge step ONLY (implementation, code, tests work without it).

**Branch:** `plan-t2-3/atlas-expo-rn` cut from `main`.

**Known v1 limitation (documented in README):** Sandbox preview serves the **web build** of the Expo app via `react-native-web`. Native iOS/Android simulator preview, deep-link / push notification / camera / native-module testing is OUT OF SCOPE for v1. Acceptable trade-off because: (a) the iframe-based E2B preview cannot host a simulator anyway, (b) the developer LLM's diff is what we care about (correct RN component tree, correct NativeWind classes, correct expo-router structure), and (c) the user can `git clone` the artifact and run `expo start --ios` / `--android` themselves once they're happy.

## Design Decisions

1. **Why Expo SDK 52 (not 51 or 53).** SDK 52 is the latest stable as of 2026-05; pairs with React Native 0.76 (the New Architecture is opt-in but stable), expo-router 4.x, and NativeWind 4.x. SDK 51 is one generation behind and would lock us out of the New Architecture toggle. SDK 53 is not yet released as of the plan-writing date. Pick the latest stable; if 53 ships before execution, bump in Task 1's `package.json` step (the rest of the plan is version-agnostic).

2. **Why expo-router 4.x (file-based) over react-navigation directly.** expo-router 4 is now the Expo team's recommended default and is what the Notion / Linear / Robinhood / Bear-style apps use. File-based routing also matches the developer LLM's intuition from the Next.js template (`app/`-directory conventions transfer almost 1:1: layouts, route groups, dynamic segments). Letting the LLM use the same mental model across `atlas-next-ts-v2` and `atlas-expo-rn` cuts the prompt-fragment surface area.

3. **Why NativeWind 4.x (not StyleSheet, not styled-components, not Tamagui).** NativeWind 4 is a strict superset of Tailwind utility classes adapted to RN, exposes `className` on RN components via a babel plugin, and is the closest mental match to the `atlas-next-ts-v2` template's Tailwind 3 setup. The developer LLM already speaks Tailwind; NativeWind makes that knowledge immediately portable. StyleSheet would mean a totally different idiom; Tamagui would mean a heavier abstraction the LLM hasn't seen as much in training data.

4. **Why the Dockerfile uses `node:22-alpine` (not `e2bdev/code-interpreter:latest`).** atlas-fastapi extended `e2bdev/code-interpreter:latest` because it needed Python + Node both. atlas-expo-rn is Node-only — no Python tooling required. Using `node:22-alpine` directly trims ~800MB off the image size and avoids inheriting the code-interpreter base's Python apt cache.

5. **Why the sandbox runs `expo start --web --port 3000` (not `--port 19006`, not Metro default 19000).** E2B's existing canvas iframe expects port 3000 (consistent with `atlas-next-ts-v2` and `atlas-fastapi`). Forcing `--port 3000` keeps `apps/atlas-web/lib/sandbox/factory.ts`'s URL-derivation logic untouched. Expo's `--web` flag boots the react-native-web bundle through Metro and serves it as a static web app, which is exactly what we want for the iframe.

6. **Why ship `app/(tabs)/_layout.tsx` even though `app/index.tsx` alone would suffice.** The file-based-routing tabs convention is the single most-used Expo Router feature. Shipping it in the template gives the developer LLM a working example to extend rather than having to invent the syntax from scratch. Keeps the LLM's diff smaller + more correct on the typical "build me a mobile app with home/settings tabs" prompt.

7. **Why `react-native-web` is implicit (auto-included by Expo).** SDK 52's web target auto-bundles react-native-web; we don't have to add it to dependencies explicitly. The Dockerfile just runs `pnpm install` and Expo wires up the web pipeline.

8. **Why `__tests__/smoke.test.tsx` uses jest-expo (not vitest).** jest-expo is the canonical RN testing harness; vitest doesn't have the RN renderer + Metro plugin setup. The smoke test asserts the root layout renders without throwing — that's enough for a sandbox template (it's not a full unit-test suite).

9. **Why per-template README documents the web-only preview limitation prominently.** Operators (and future maintainers) need to know upfront that the iframe preview shows web, not native. Burying it in a footnote risks someone shipping a native-only feature (e.g., `expo-camera`) and being surprised the preview doesn't work.

10. **Why we hand-author `pnpm-lock.yaml` rather than committing `package-lock.json`.** Atlas monorepo standard is pnpm 9 (per memory + Plan T.1). `pnpm install` inside the Dockerfile produces a deterministic `node_modules` from the lockfile; committing the lockfile pins all transitive deps cryptographically.

11. **Why update the `mobile-app` route case in template-router (instead of leaving the fallback).** T.1 explicitly leaves `mobile-app` as a fallback to `atlas-next-ts-v2` because no template existed yet. This plan ships that template. Now `templateForArtifactKind("mobile-app", { multiStackFlagOn: true })` should return `"atlas-expo-rn"`. The template-router test gets a corresponding update.

12. **Why a new researcher skill (`assemble-brief-mobile-app.md`) rather than reusing the generic one.** T.1's pattern: per-`artifactKind` skills cite references appropriate to the category. For mobile, that means iOS / Android consumer-app references (Notion mobile, Linear mobile, Things 3, Bear, Robinhood) — not landing pages, not REST APIs. Without this skill, the researcher would cite linear.app's marketing site (a landing page) when asked about Linear's mobile app — wrong target.

13. **Why ship the v1 template-id as `<TBD-after-build>` placeholder.** Same pattern as T.1 — implementing engineer commits the e2b.toml without a real template_id; operator post-merge runs `build-template.sh`, captures the printed ID, commits a one-line follow-up updating `template_id`. Avoids embedding a fake ID that will silently break.

14. **Why no atlas-web factory changes in this plan.** T.1 already wired `resolveTemplateForRitual` in `apps/atlas-web/lib/sandbox/factory.ts` to call `templateForArtifactKind(...)`. Since this plan only changes what `templateForArtifactKind` returns for `mobile-app`, no factory wiring changes needed. The integration test from T.1 (`factory-multi-stack.test.ts`) still passes; we add a new case for mobile-app in this plan's Task 9.

```dockerfile
# atlas-expo-rn: Expo SDK 52 + RN 0.76 + expo-router 4 + NativeWind 4 web build
# served at port 3000 for the E2B preview iframe. Native iOS/Android simulator
# preview is NOT supported in v1 — sandbox runs `expo start --web` only.
FROM node:22-alpine

# pnpm + system deps Expo / Metro need (git for some plugin install paths,
# bash for run scripts, libc6-compat for native modules built against glibc).
RUN apk add --no-cache git bash libc6-compat curl ca-certificates \
 && corepack enable \
 && corepack prepare pnpm@9.12.0 --activate

# Atlas convention: app code lives at /code, owned by `user`.
RUN addgroup -S user && adduser -S user -G user
WORKDIR /code

# Copy manifests first for cache-friendly layering.
COPY package.json ./
COPY pnpm-lock.yaml* ./

# Install deps.
RUN pnpm install --frozen-lockfile 2>/dev/null || pnpm install

# Copy the rest of the template.
COPY tsconfig.json app.json babel.config.js metro.config.js tailwind.config.js global.css nativewind-env.d.ts ./
COPY app ./app
COPY components ./components
COPY __tests__ ./__tests__

# Pre-warm the Expo + Metro bundler caches so the first sandbox boot is fast.
# `expo prebuild --no-install` would generate native projects; we skip that —
# we only ever run the web build. Instead, we eagerly run `expo start --web`
# in a build-time dry pass to populate Metro's cache directory.
RUN chown -R user:user /code

USER user

# Assemble Brief — Mobile Apps

Use this fragment when `designIntent.artifactKind === "mobile-app"`. The artifact is a native (or react-native-web) mobile app, not a marketing landing page and not a desktop web app. Your reference picks must be **mobile apps** specifically — most peer products have both web and native, and the mobile UX is materially different.

## Reference catalog (cite at least one peer in the user's specific category)

- **Notion mobile** — clean text-first iOS/Android, custom block editor, tab-bar navigation, dark/light auto-switch. Best peer for "knowledge management / docs" categories.
- **Linear mobile** — keyboard-shortcut-equivalent gestures, command-palette via long-press, immediate optimistic UI, tab-bar navigation. Best peer for "team productivity / project tracking" categories.
- **Things 3 (iOS)** — gold standard for tasks/todo apps; magic-plus button, swipe-to-schedule, today-vs-upcoming-vs-anytime mental model. Best peer for "personal productivity / habits / todo".
- **Bear (iOS/Mac)** — plain-text + markdown, beautiful typography, hashtag-based organization, no folders. Best peer for "writing / journaling / notes".
- **Robinhood** — number-heavy financial dashboards, tab navigation, gesture-rich charts, dark-by-default. Best peer for "money / fintech / dashboards-with-numbers".
- **Apple's stock apps** (Mail, Calendar, Notes, Reminders) — set the platform-native bar; cite when the user wants "feels like an Apple app."
- **Spotify mobile** — tab-bar + tall scroll cards + persistent bottom mini-player. Best peer for "media playback / library / audio".
- **Instagram / TikTok** — only cite for short-form-video / social-feed categories; otherwise their UI patterns will pull the design in the wrong direction.

## Quality bar

- Headlines name a concrete user task ("Track your habits", "Capture every idea"), not "experience the future of mobile."
- Hero / first-screen sketch must show a real-feeling RN component tree — `<Tabs>` at the bottom OR `<Stack>` with a single screen, not a marketing page.
- Cite at least one peer **mobile app** in the user's specific category, not just "all mobile apps."
- Touch targets ≥ 44pt (iOS HIG), bottom-tab bar height ≈ 49pt + safe-area inset.
- Default to platform conventions: iOS uses tab bars at the bottom; Android uses bottom-nav OR drawer; both look fine on react-native-web.
- Dark/light mode awareness: cite the peer's dark-mode treatment if relevant.

## Anti-patterns

- Don't propose a marketing landing page when the artifact is the app itself.
- Don't reach for desktop-style sidebars; mobile screens are narrow.
- Don't propose modal-on-modal stacks; iOS HIG limits to one modal layer in most flows.
- Don't propose web-only widgets (hover tooltips, right-click menus, drag-and-drop kanban) — they don't translate to touch.
- Don't cite linear.app / notion.so (their *marketing sites*) — cite the *apps themselves* (Linear iOS, Notion iOS).
- Don't assume the user wants Material Design specifically; default to platform-native (iOS HIG on iOS, Material on Android — Expo handles both).

## Token-budget guidance

Keep brief ≤ 1.5KB. Cite 2-3 references max — more dilutes signal. Researcher's job is to pick the right peer and articulate WHY it's the peer, not enumerate all mobile apps.
```
