## Design Decisions

1. **Why Bun (not Node) as runtime + package manager + test runner.** Bun's HTTP server (`Bun.serve`) is ~3-5× faster than Node's, has first-class TypeScript support (no transpile step at boot), and bundles a test runner (`bun:test`). For a GraphQL server template that boots inside an E2B sandbox, every second of cold start matters — Bun gets us from "user submits prompt" to "GraphQL endpoint live" in <2 seconds. ADR-001 explicitly endorses Bun for new TypeScript-runtime work.

2. **Why GraphQL Yoga (not Apollo Server, not graphql-helix, not graphql-http).** Yoga 5 is the GraphQL-The-Guild successor to graphql-yoga v3 + Apollo Server merged: tiny dependency footprint, Envelop plugin system for auth/persisted-queries/tracing, **first-class Bun.serve compatibility via the `createYoga` handler returning a fetch-style `Request → Response` handler** (no Express/Koa adapter needed). Apollo Server v4 works with Bun but ships way more deps. graphql-helix is unmaintained. graphql-http is too low-level (you'd build your own GraphiQL).

3. **Why Pothos (code-first, not SDL-first).** Code-first means the schema is derived from TypeScript types, so the type-safety is end-to-end (resolver arg types ⇒ schema types ⇒ generated client types). SDL-first (schema.graphql + codegen) doubles the surface area, requires a watcher, and never quite types the resolvers as tightly. Pothos 4.x is the de-facto code-first builder in 2026 — actively maintained, plugin ecosystem (Drizzle, Prisma, Auth, Federation, Smart-Subscriptions) covers everything we'd want. Per the spec: "Pothos GraphQL 4.x (code-first schema builder; preferred over SDL-first)".

4. **Why `@pothos/plugin-drizzle` (not `@pothos/plugin-prisma`).** ADR-001 picks Drizzle over Prisma for the Atlas backend (smaller bundle, no separate `prisma generate` step, SQL-first ergonomics for diego/priya users who want to read the queries). The Pothos Drizzle plugin generates the right `t.relation()` / `t.field()` resolvers with N+1 batching automatically — same DX as the Prisma plugin without the heavyweight client.

5. **Why GraphiQL only in dev.** Yoga ships GraphiQL out of the box; we leave it on (matches the FastAPI template's `/docs` Swagger UI design — gives diego/priya users an immediate "this is real" moment). Production templates deviate from this (auth-gated GraphiQL, persisted queries only) but a sandbox template is by definition development-only.

6. **Why a smoke `/health` endpoint AND the GraphQL endpoint.** `/health` is the canonical Kubernetes readiness check pattern, used by the e2b.toml `ready_cmd` to know when the sandbox is up. The GraphQL endpoint is auth-gated by default (rate-limited, etc.) in real apps; the `/health` route is plain JSON, fast, idempotent — ideal for readiness.

7. **Why `bun:test` (not vitest/jest).** Bun's test runner is the native choice when the runtime is Bun; matches `bun:sqlite`, `bun:ffi`, etc. naming. No additional dev-dep bloat (`bun:test` is built into the Bun binary). Other packages in the repo (atlas-web, role-developer) keep using vitest because they target Node/Next.js — the template directory is its own world.

8. **Why route `backend-graphql` (not introduce a new artifactKind).** The architect's `ArtifactKindSchema` already includes `"backend-graphql"`. The T.1 router has a `case "backend-graphql":` that currently falls through to `DEFAULT_TEMPLATE`. We replace the fallback with the real template — zero schema changes.

9. **Why per-template README.** Same as T.1 — each template's quirks (boot command, dep install, smoke-test URL) differ. Co-locating docs with the template is the easiest pattern for operators.

10. **Why ship the v1 template-id as `<TBD-after-build>` placeholder.** Same pattern as Plan S.1 + T.1 — implementing engineer commits the e2b.toml without a real template_id; operator post-merge runs `build-template.sh`, captures the printed ID, commits a one-line follow-up updating `template_id`. Avoids embedding a fake ID that will silently break.

11. **Why no Drizzle DB connection in the v1 template.** v1 ships the *schema-builder plugin* loaded but no real database connection — the template's `src/schema/index.ts` only has a `hello: String!` field. Adding Postgres / Neon / SQLite at the template level forces the operator to ship secrets per-sandbox. The Pothos+Drizzle plugin is *importable* and the prompt fragment teaches the developer to use `t.relation()` / `t.field({ type: ... })` — the actual `drizzle()` connection is something the developer's diff will add when the user requests a feature that needs persistence (matches the `atlas-fastapi` pattern of shipping SQLAlchemy installed but not wired to a DB).

12. **Why Researcher skill cites Hasura / GitHub / Linear / Shopify (not Apollo / Relay).** Hasura is the ur-example of "GraphQL as a tier-0 backend" — diego/priya users have heard of it. GitHub, Linear, Shopify are the three most-cited public GraphQL APIs in 2026 (hierarchical resources, mutation patterns, pagination via cursors, error envelopes). Apollo's docs are framework-specific; Relay is client-side. The Researcher's job is to cite *peer products* the user can compare to, not framework docs.

13. **Why update `assemble-prompt.test.ts` (not just registry test).** The registry test asserts the `getSandboxContextPrompt("atlas-graphql-yoga")` lookup directly; the assemble-prompt test asserts that `getSandboxContextPromptFor("atlas-graphql-yoga")` (the public re-export) returns the same prompt. Both are needed because they exercise different module boundaries.

```ts
import { createYoga } from "graphql-yoga";
import { schema } from "./schema/index.js";

const yoga = createYoga({
  schema,
  graphqlEndpoint: "/graphql",
  // GraphiQL on by default in dev. To disable, pass `graphiql: false`.
  graphiql: { title: "atlas-graphql-yoga" },
  // Yoga handles CORS itself; relax for local dev.
  cors: { origin: "*", credentials: false },
  context: ({ request }) => ({ request })
});

export interface CreateServerOptions {
  port?: number;
  hostname?: string;
}

export function createServer(opts: CreateServerOptions = {}) {
  const port = opts.port ?? Number(process.env.PORT ?? 3000);
  const hostname = opts.hostname ?? "0.0.0.0";

  return Bun.serve({
    port,
    hostname,
    fetch: async (request) => {
      const url = new URL(request.url);

      if (request.method === "GET" && url.pathname === "/health") {
        return Response.json({
          status: "ok",
          stack: "graphql-yoga",
          atlas: "sandbox-ready"
        });
      }

      if (request.method === "GET" && url.pathname === "/") {
        return Response.json({
          name: "Atlas Sandbox",
          version: "0.1.0",
          graphqlEndpoint: "/graphql"
        });
      }

      // Yoga handles /graphql (POST queries + GET GraphiQL).
      return yoga.fetch(request);
    }
  });
}

// Boot when invoked directly (Bun.main pattern).
if (import.meta.main) {
  const server = createServer();
  console.log(
    `atlas-graphql-yoga ready on http://${server.hostname}:${server.port} ` +
      `(GraphQL @ /graphql, GraphiQL @ /graphql)`
  );
}
```

FROM e2bdev/code-interpreter:latest

RUN apt-get update \
 && apt-get install -y --no-install-recommends curl ca-certificates unzip \
 && rm -rf /var/lib/apt/lists/* \

 && cp /root/.bun/bin/bun /usr/local/bin/bun \

COPY src/server.ts ./src/
COPY src/schema/builder.ts src/schema/index.ts ./src/schema/
COPY tests/server.test.ts ./tests/

# --frozen-lockfile keeps the operator's lockfile authoritative; on first
# build (no lockfile present) bun installs without --frozen.
RUN bun install --frozen-lockfile 2>/dev/null || bun install \
 && chown -R user:user /code

USER user

EXPOSE 3000

# Note: e2b.toml's start_cmd overrides this CMD at sandbox boot.
CMD ["bun", "run", "src/server.ts"]

# Assemble Brief — Backend GraphQL APIs

Use this fragment when `designIntent.artifactKind === "backend-graphql"`. References are GraphQL API products and well-known public schemas — Hasura, GitHub GraphQL API, Linear API, Shopify Storefront API, Stripe (GraphQL preview), Contentful, GraphCMS / Hygraph. NOT REST API products (those are for `backend-rest-api`); NOT landing pages (those are `frontend-app`).

## Quality bar

- **Schema design follows graph-shape conventions.** Plural query roots return `Connection`-style cursor pagination (`edges { node { ... } cursor } pageInfo`), not bare arrays. Singular fetches by ID use `Node`-interface conformance where useful (Relay-style global IDs).
- **Mutations follow the input-payload pattern.** Inputs go through a single `input: SomethingInput!` argument; payloads return both the mutated entity and a `userErrors` array (Shopify-style — far more useful than throwing GraphQL errors for validation failures).
- **Errors split into "system errors" (GraphQL `errors` array — auth, rate-limit, internal) vs. "domain errors" (`userErrors` field on mutation payloads — validation, business rules).** The Researcher MUST cite at least one peer schema (Shopify, Linear) that follows this split.
- **Auth uses standard patterns.** Bearer token in `Authorization` header (GitHub, Linear) OR API key in `X-API-Key` (Stripe-style). Do NOT invent auth headers.
- **Pagination MUST be cursor-based, not offset-based** for any unbounded list. Cite Relay's connection spec.
- **N+1 batching is non-negotiable.** Cite Pothos's Drizzle plugin or DataLoader as the chosen mechanism.
- **Multi-tenancy is explicit.** Tenants table + RLS, OR a `viewer` field on the root Query that scopes child resolvers to the current actor (GitHub-style `viewer { repositories { ... } }`).
- **Smoke-test field present.** Schema MUST include a `hello: String!` (or similar) trivial query field that proves the server is up — matches the `/health` REST pattern.

## Anti-patterns

- Don't propose RPC-style fields (`doSomething: Boolean`) when graph-shape resolves the same intent (`updateOrder(input: ...): UpdateOrderPayload`).
- Don't return raw arrays for list fields. Always use `Connection`-shape pagination.
- Don't return `errors: [String]` from mutation payloads — use a typed `userErrors: [UserError!]!` with `field: [String!]`, `code: ErrorCode!`, `message: String!`.
- Don't invent custom scalars (`DateTime`, `UUID`) without citing where they're defined. Use Pothos's built-in scalars or import from `graphql-scalars`.
- Don't ship a schema with mutations but no `input` types — every mutation must take a single `input: <Mutation>Input!` argument for forward compatibility.
- Don't ship without subscriptions when the user prompt mentions "live", "real-time", "feed", "notifications" — subscriptions are GraphQL's superpower; missing them is a missed opportunity.
- Don't propose Apollo Federation for a 1-service project (overkill).

## Reference schemas to cite

When the user's project domain matches, cite by name in the Researcher brief:

- **Marketplace / e-commerce** → Shopify Storefront API (cart, checkout, products, customer)
- **Issue tracker / project management** → Linear API (Issue, Project, Cycle, Comment)
- **Code hosting / SCM** → GitHub GraphQL API (Repository, PullRequest, Issue, viewer)
- **Headless CMS** → Contentful, Hygraph (content models, locales, asset types)
- **Payments (preview)** → Stripe GraphQL API patterns
- **Generic CRUD-over-Postgres** → Hasura's auto-generated schemas (insert/update/delete by PK, where clauses)

## Output style

The Researcher brief should:
1. Open with the chosen reference schema(s) and one paragraph of WHY (2-3 sentences max).
2. List 5-8 root Query fields for the user's domain, naming the return type as a `Connection` or specific object.
3. List 3-5 mutations with their `input` shapes and `userErrors` codes.
4. Note any subscription opportunities ("real-time order status" → `subscription orderStatusUpdated(orderId: ID!): OrderStatusEvent!`).
5. Close with auth scheme + multi-tenancy approach.
