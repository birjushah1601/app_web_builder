**Why a status page (and not a web UI)?** CLIs don't have a web UI. The sandbox preview iframe still expects *something* on port 3000 (the canvas's "preview unavailable" red card otherwise greets every CLI project). The pragmatic v1 choice is to bundle a tiny Bun.serve status page that renders the CLI's name + version + a one-line "run `atlas help` in the terminal" hint + an ascii-art logo. The CLI itself is exercised by Atlas's Developer role via the existing E2B Exec API (`sandbox.commands.run("bun run src/cli.ts hello --name world")`); the iframe is just a "sandbox is alive" indicator. A v2 polish lands an xterm.js-based terminal-emulator viewer that live-streams the CLI's last invocation output via WebSocket — nice but not required for v1.

## Design Decisions

1. **Why Bun (not Node + tsx + npm).** Single binary download, runs `.ts`/`.tsx` natively (no esbuild/tsx wrapper), 4-10× faster install than npm/pnpm, has a built-in test runner (`bun:test`) and a built-in HTTP server (`Bun.serve`). For a CLI sandbox the developer-feedback loop has to be near-zero; Bun's cold-start and `bun --hot` reload time fits that. Bun also ships `bun build --compile` which produces a single-file binary — useful for the v2 "publish your CLI" story. Node would mean shipping `tsx` for the dev loop and `pkg`/`esbuild` for binaries — strictly worse for this use case.

2. **Why Commander 12.x (not Yargs / oclif / clipanion).** Commander is the de-facto standard for Node/Bun CLIs (used by gh CLI's Node bits, vercel CLI, and dozens of OSS tools), has a tiny API surface (`program.command(...).description(...).action(...)`), and ships its own help-output generator. Yargs is more configurable but its DSL is more bespoke. oclif is heavier (plugin system, manifest generation) — overkill for v1 sandboxes. clipanion is excellent but niche. Commander is the lowest-surprise pick.

3. **Why ink 5.x (React for CLIs).** Two reasons. **(a)** Atlas's Developer role already speaks JSX (Next.js + React). Reusing JSX for interactive CLI screens means the Developer doesn't have to learn a separate terminal-UI DSL — `<Box>`, `<Text color="cyan">`, `<Spinner>` map cleanly to mental models the role already has. **(b)** ink has a real ecosystem (`ink-spinner`, `ink-text-input`, `ink-select-input`, `ink-table`, `ink-progress-bar`, `ink-box`) so even small ink components feel polished. The alternative — raw `process.stdout.write` + chalk + readline — works for trivial CLIs but degrades rapidly when interactivity is needed.

4. **Why chalk 5.x (and not picocolors).** ink ships chalk transitively for its `color` prop logic. Adding chalk explicitly lets non-ink code paths (early-exit error printing, plain-text output piped to other commands) use the same color primitives. picocolors is smaller but doesn't pair as naturally with ink's `<Text color>`.

5. **Why zod for input validation.** Commander's `--option <value>` parsing is loose (everything is string by default; `.argParser(parseInt)` is per-arg). For CLIs that take JSON-shaped input (e.g., a `provision` command that reads a config file), zod is the standard validation layer the developer prompt already references in the `next-ts-v2` and `fastapi` templates. Keeping it consistent across templates means the developer's "I trust zod" muscle memory carries over.

6. **Why bun:test (not vitest).** Bun's built-in test runner has zero install cost (already inside the runtime), supports the same `describe / it / expect` API vitest does, and runs `.ts`/`.tsx` natively. Adding vitest would mean adding `vitest`, a `vitest.config.ts`, and the Vite toolchain — wasted bytes when the runtime ships a perfectly good runner.

7. **Why the "Bun.serve status page" preview surface (and not xterm.js).** Pragmatism. The canvas iframe expects port 3000 to respond with HTML. The status page is ~30 lines of HTML + ascii-art served by `Bun.serve` — finishes in one task; works the moment the sandbox boots. The xterm.js terminal emulator viewer (live-streams CLI output via WebSocket) is the *right* long-term answer but adds: (a) WebSocket plumbing, (b) a process-output multiplexer, (c) xterm.js client bundle, (d) auth on the WebSocket, (e) tests for all of it. That's a v2 plan, not a v1 task. Status page now, terminal viewer later.

8. **Why `src/commands/<name>.tsx` convention (and not `src/cli.ts` monolith).** Scales cleanly. As soon as a CLI has 3+ subcommands, a single file becomes a 500-line hairball. The `src/commands/<name>.tsx` convention mirrors Next.js App Router and Express's per-route file convention — both Atlas already speaks. Each subcommand file exports a default ink component + a `register(program)` function; `cli.ts` does nothing but import + register them.

9. **Why `oven/bun:1-alpine` base image.** Smallest official Bun image (~50 MB compared to `oven/bun:1` Debian-based ~150 MB). Alpine's musl libc occasionally trips up native modules but Commander/ink/chalk/zod are pure JS — no native deps to worry about. If a sub-plan ever needs `bun add sharp` or similar, switching to `oven/bun:1` is one line.

10. **Why ship the v1 template-id as `<TBD-after-build>` placeholder.** Identical pattern to T.1 / T.2.x — implementing engineer commits the e2b.toml without a real template_id; operator post-merge runs `build-template.sh`, captures the printed ID, commits a one-line follow-up updating `template_id`. Avoids embedding a fake ID that will silently break.

11. **Why update `template-router.ts`'s `cli-tool` case (not add a new branch).** T.1 already defined the case with a v2-deferred fallback to `atlas-next-ts-v2`. The change is one line — replace the fallback's return value. Tests update accordingly. Keeps the diff tight.

12. **Why a researcher skill `assemble-brief-cli-tool.md`.** When a user prompts "build me a CLI for batch-renaming photos", the Researcher needs to cite *CLI design patterns*, not landing-page references. The skill cites: gh CLI (subcommand grouping, `gh repo create` patterns), kubectl (verb/noun structure: `get`, `apply`, `delete`), fly CLI (provisioning UX with status spinners), npm CLI (everyone's baseline), vercel CLI (interactive prompts done well), bun CLI itself (Bun's own CLI for self-reference), and three ink-based OSS examples (`pastel`, `tasktimer`, `terminal-image-cli`) for ink-specific patterns. The Researcher's `RoleInvocation` already has the per-artifactKind skill composition logic from T.1 — this just adds the file.

```ts
#!/usr/bin/env bun
import { Command } from "commander";
import { registerHello } from "./commands/hello.js";

/**
 * Build the root Commander program. Exported so tests can introspect it
 * without invoking process.exit. The actual entry-point at the bottom calls
 * `program.parseAsync(process.argv)`.
 */
export function buildProgram(): Command {
  const program = new Command();

  program
    .name("atlas")
    .description("Atlas CLI — generated by the Atlas builder (cli-tool artifactKind)")
    .version("0.1.0");

  registerHello(program);

  return program;
}

// Run only when invoked as a binary (not when imported by tests).
if (import.meta.main) {
  const program = buildProgram();
  await program.parseAsync(process.argv);
}
```

```tsx
import { Command } from "commander";
import React, { useEffect, useState } from "react";
import { render, Box, Text } from "ink";
import Spinner from "ink-spinner";

interface HelloProps {
  name: string;
}

/**
 * Default-export ink component for the `hello` subcommand. Demonstrates the
 * canonical ink pattern: <Box> for layout, <Text color="..."> for color,
 * <Spinner> for in-flight indicators, hooks for state.
 */
export default function Hello({ name }: HelloProps): React.ReactElement {
  const [done, setDone] = useState(false);

  useEffect(() => {
    const t = setTimeout(() => setDone(true), 600);
    return () => clearTimeout(t);
  }, []);

  if (!done) {
    return (
      <Box>
        <Text color="cyan">
          <Spinner type="dots" />
        </Text>
        <Text> Greeting {name}…</Text>
      </Box>
    );
  }

  return (
    <Box flexDirection="column">
      <Text color="green">✓ Hello, {name}!</Text>
      <Text dimColor>(rendered via ink — Atlas CLI v0.1.0)</Text>
    </Box>
  );
}

/**
 * Register the `hello` subcommand on a Commander program. Convention for all
 * subcommand files: export `registerX(program: Command)`. cli.ts wires them
 * up; tests can register against an isolated program for assertions.
 */
export function registerHello(program: Command): void {
  program
    .command("hello")
    .description("Print a friendly greeting (ink demo)")
    .option("-n, --name <name>", "name to greet", "world")
    .action((opts: { name: string }) => {
      render(<Hello name={opts.name} />);
    });
}
```

```ts
/**
 * Start the status-page HTTP server. Returns the Bun.Server handle so callers
 * (tests, the Dockerfile entrypoint) can stop it. Default port 3000 matches
 * the e2b.toml ready_cmd / start_cmd.
 */
export function startServer(opts: { port?: number } = {}): Server {
  const port = opts.port ?? 3000;
  return Bun.serve({
    port,
    fetch(req) {
      const url = new URL(req.url);
      if (url.pathname === "/health") {
        return Response.json({ status: "ok", stack: "bun-cli", atlas: "sandbox-ready" });
      }
      return new Response(STATUS_HTML, {
        status: 200,
        headers: { "content-type": "text/html; charset=utf-8" }
      });
    }
  });
}

if (import.meta.main) {
  const server = startServer({ port: Number(process.env.PORT ?? 3000) });
  console.log(`atlas-bun-cli status page → http://localhost:${server.port}/`);
```

```markdown
# Assemble Brief — CLI Tools

Use this fragment when `designIntent.artifactKind === "cli-tool"`. Citations are well-known terminal CLIs and ink-based OSS examples — NOT landing pages, NOT API docs.

## Reference CLIs to cite for command-design patterns

- **gh CLI (cli/cli)** — gold standard for resource-oriented subcommand grouping (`gh repo create`, `gh pr list`, `gh issue close`). Hierarchical noun-then-verb structure scales to dozens of subcommands without users getting lost.
- **kubectl** — verb-first action vocabulary (`get`, `apply`, `delete`, `describe`, `logs`). Pair with consistent resource nouns (`pod`, `service`, `deployment`). Ideal when the CLI has many actions across few resource types.
- **fly CLI (flyctl)** — exemplary provisioning UX: status spinners while infra spins up, real-time progress prints, color-coded status (green check / yellow spinner / red X), terse but informative error messages with remediation hints.
- **npm CLI** — everyone's baseline; conservative output by default, `--json` flag for machine-readable, `--verbose` for debugging. Three output modes is plenty.
- **vercel CLI** — best-in-class interactive prompts when arguments are missing (project pick, env-var input, deploy confirmation). Use `ink-text-input` and `ink-select-input` to mirror this pattern.
- **bun CLI itself** — Bun's own command surface (`bun run`, `bun test`, `bun install`, `bun build`). Self-referential; read it for "what does a Bun-native CLI feel like" answers.

## Reference ink-based CLIs to cite for terminal-UI patterns

- **pastel** (vadimdemedes/pastel) — Next.js-style file-router framework for ink CLIs by ink's author. Read it to understand the conventional shape of a multi-screen ink app.
- **tasktimer** (rasaf-ibrahim/tasktimer) — small, polished ink CLI; good reference for a single-screen interactive UI with timer + keyboard handling.
- **terminal-image-cli** (sindresorhus/terminal-image-cli) — minimal, clean Commander + render-to-stdout pattern; good baseline for "no interactivity, just print well-formatted output".

## Quality bar

- **Subcommand grouping is intentional.** Either resource-noun-first (gh / vercel) OR verb-first (kubectl / npm) — pick one model and stick to it. Mixing the two confuses users.
- **Help output is readable.** Commander auto-generates `--help`; the description strings the developer writes ARE the docs. Cite this in the brief.
- **Default output is terse.** Use `--verbose` / `-v` for debug noise; never default to chatty.
- **Errors have remediation.** Don't print `Error: file not found`. Print `Error: file not found at <path>. Did you mean to run \`atlas init\` first?`.
- **Color is meaningful, not decorative.** Green = success / present; red = error / destructive; yellow = warning / in-progress; cyan = informational header. Cite chalk's role for non-ink output.
- **Interactive prompts only when arguments are missing.** Honour `--no-interactive` / `--yes` flags. Cite vercel's pattern.
- **Exit codes are deliberate.** 0 = success, 1 = generic failure, 2 = misuse (wrong args). Document the convention in `--help`.

## Anti-patterns

- Don't propose a CLI that requires a long-running daemon — keep the command surface request-response.
- Don't reach for ink for one-shot output (a `print version` doesn't need React) — chalk + `console.log` is fine; reserve ink for interactivity / multi-line live updates.
- Don't invent flag names that conflict with conventions — `--help`, `--version`, `--verbose`, `--quiet` MUST behave the way users expect.
- Don't ship a CLI without `--version` (the command should exit 0 even if no other config is set up).
- Don't write the CLI as if it's a web app — there's no port to bind to (the status page on port 3000 is sandbox plumbing, not part of the user's CLI).
```
