# Plan T.2.1 — atlas-hono-bun (Bun + Hono + Drizzle ORM, alternative `backend-rest-api`) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Add a Bun + Hono + Drizzle ORM E2B template (`atlas-hono-bun`) as a TypeScript-native alternative to `atlas-fastapi` for the `backend-rest-api` artifact kind. Register a new `hono-bun` developer prompt fragment in the per-template registry shipped by Plan T.1. Behind the same `ATLAS_FF_MULTI_STACK` flag — no new flag added in this plan. Users opt into Hono per-project via `ATLAS_DEFAULT_SANDBOX_TEMPLATE=atlas-hono-bun`; T.1's `templateForArtifactKind` continues to route `backend-rest-api → atlas-fastapi` as the default. A future v2 architect signal (e.g., a `runtimePreference: "node-typescript"` hint inside `canvasManifest`) can flip the routing to Hono on a per-project basis without touching this plan's deliverables.

**Architecture:** This plan is a thin, mechanical follow-up to T.1 — the registry, flag, router, and factory plumbing already exist. Three coordinated changes. **(1)** New `packages/sandbox-e2b/templates/atlas-hono-bun/` mirroring the atlas-fastapi shape — `package.json` + `tsconfig.json` + `Dockerfile` (oven/bun:1-alpine) + `e2b.toml` + minimal Hono app with `GET /` + `GET /health` smoke endpoints + `bun:test` smoke tests + scripts + README. Built + pushed to E2B by an operator post-merge (mirrors T.1's deferred republish). **(2)** New `packages/role-developer/src/sandbox-context-prompts/hono-bun.ts` positive-list prompt fragment + registration in `sandbox-context-registry.ts` + tests covering the lookup. **(3)** Comment-only nudges in `apps/atlas-web/lib/sandbox/template-router.ts` + `apps/atlas-web/.env.example` documenting that `atlas-hono-bun` is a valid `ATLAS_DEFAULT_SANDBOX_TEMPLATE` value for users who prefer TypeScript backends. NO behaviour change to the router default — backend-rest-api still routes to atlas-fastapi unless the user pins.

**Tech Stack:** Bun 1.2+ (runtime + package manager + test runner) · Hono 4.x (web framework) · Drizzle ORM 0.36+ (with `postgres-js` driver; `@libsql/client` is a documented opt-in for embedded use) · Zod 3.x (request/response validation; already in workspace) · TypeScript 5.6 strict · `bun:test` for tests · E2B SDK + CLI · Docker.

## Design Decisions

1. **Why ship Hono + Bun alongside FastAPI (not as a replacement).** FastAPI matches Python-fluent users (diego/priya in legacy SaaS shops). Hono + Bun matches TypeScript-fluent teams (faster cold-start, single language across frontend + backend, smaller image). Same artifact kind (`backend-rest-api`); user picks the preferred runtime per project via env override. v1 of T (T.1) routed backend-rest-api → atlas-fastapi as the safer default — Python is the most-requested REST stack in our user research. T.2.1 keeps that default; Hono is opt-in.

2. **Why Bun (not Node).** Bun 1.2+ is production-stable (2025), ships a fast test runner (`bun:test`), TypeScript transpilation built in (no `tsx` / `ts-node` needed), and a single binary install — much simpler Dockerfile than Node + corepack + pnpm. The bundle-vs-runtime story stays compatible with Hono (Hono's runtime-agnostic design works on Bun, Node, Deno, Cloudflare Workers).

3. **Why Hono (not Express, not Fastify).** Hono 4.x is the smallest API surface for typed routes. Its built-in Zod helpers (`@hono/zod-validator`) match the existing workspace Zod 3.x dep, so we don't introduce a parallel validation library. Express is older + uses callback-style middleware that fights TS inference; Fastify's plugin system is heavier than what a sandbox needs.

4. **Why Drizzle (not Prisma).** Drizzle is SQL-first, no codegen step, no separate migration runner daemon. Prisma's `prisma generate` adds a build-time step that breaks Bun's "just run it" model and adds 100MB+ to the image. Drizzle's `drizzle-kit` is a CLI-only tool, ships nothing into the runtime — keeps the sandbox image lean. The `drizzle-orm/postgres-js` driver is the standard for serverful Postgres; `@libsql/client` is a documented opt-in for embedded SQLite (mentioned in README, not installed by default).

5. **Why `oven/bun:1-alpine` base (not e2bdev/code-interpreter:latest like FastAPI).** The atlas-fastapi Dockerfile starts from `e2bdev/code-interpreter` because Python tooling is heavy and the e2b base ships system Python. For Bun, the e2b base wastes ~400MB on Python deps we don't need. `oven/bun:1-alpine` is ~50MB and starts fast. We do still need to layer e2b's hot-reload / file-sync hooks at sandbox boot — handled by the e2b runtime overlay, not the base image.

6. **Why `bun:test` (not vitest, not jest).** It's built into the runtime; no extra install. Compatible with Jest's expect API. One less dep to keep in sync with workspace versions. Keeps the sandbox image minimum.

7. **Why register the prompt fragment in the existing T.1 registry (not extend the registry data shape).** The registry is a `Record<string, string>` keyed by template name. Adding a new entry is a one-line change — no schema change, no migration, no consumer update needed downstream. T.1's `getSandboxContextPrompt(name)` already handles unknown templates via graceful fallback to default.

8. **Why no router default change (backend-rest-api still routes to atlas-fastapi).** Behavioural lock: T.1 set the default; T.2.1 should not change established routing in a follow-up template plan. Users wanting Hono pin per project. A future plan (T.3 — "architect runtime hint") can add a structured signal that flips routing without env override; that's out of scope for T.2.1.

9. **Why `package.json` + `bun.lockb` (not pnpm-workspace member).** The template directory is NOT part of the workspace — it's an isolated artifact published to E2B. Workspace inclusion would mean every `pnpm install` at repo root tries to install Bun-specific deps (postgres-js, drizzle-orm) into the root `node_modules`, which conflicts with apps/atlas-web's exact-version pins. Keeping the template's `package.json` standalone (with its own `bun.lockb`) preserves the same isolation atlas-fastapi has via its own `pyproject.toml` + `uv.lock`.

10. **Why ship example route + Drizzle schema scaffolding (vs. bare /health only).** The atlas-fastapi template's `app/main.py` ships only `/`, `/health`, `/docs`. For Hono, the developer LLM needs *style guidance* — "where do I put new routes?" — and a one-file example saves dozens of round-trips with the LLM mis-guessing the layout. The example route is commented-out by default so it doesn't pollute the smoke endpoints; the README documents how to enable it.

11. **Why ship the v1 template_id as `<TBD-after-build>` placeholder.** Same pattern as Plan T.1 / S.1 — implementing engineer commits the e2b.toml without a real template_id; operator post-merge runs `build-template.sh`, captures the printed ID, commits a one-line follow-up updating `template_id`. Avoids embedding a fake ID that will silently break.

```ts
import { Hono } from "hono";
import { zValidator } from "@hono/zod-validator";
import { z } from "zod";

/**
 * Example feature router. NOT mounted by default — uncomment the
 * `app.route("/example", exampleRouter)` line in src/index.ts to enable.
 *
 * Demonstrates the pattern Atlas's developer role should use when adding
 * new features:
 *   1. Define a Hono router scoped to the feature.
 *   2. Use `@hono/zod-validator` for typed request bodies / queries / params.
 *   3. Return JSON via `c.json(...)`.
 *   4. Throw `HTTPException` (from `hono/http-exception`) for HTTP errors.
 */
const exampleRouter = new Hono();

const CreateItemSchema = z.object({
  name: z.string().min(1).max(120),
  quantity: z.number().int().min(0),
});

exampleRouter.get("/items", (c) =>
  c.json({ items: [] as Array<{ id: string; name: string; quantity: number }> })
);

exampleRouter.post(
  "/items",
  zValidator("json", CreateItemSchema),
  (c) => {
    const input = c.req.valid("json");
    return c.json(
      { id: crypto.randomUUID(), ...input },
      201
    );
  }
);

export default exampleRouter;
```

# atlas-hono-bun: Bun 1.2 + Hono + Drizzle dev sandbox for Atlas live preview.
FROM oven/bun:1-alpine

# E2B's runtime overlay expects a `user` user with /code workdir + write perms.
RUN addgroup -S user && adduser -S -G user user \
 && mkdir -p /code \
 && chown -R user:user /code

WORKDIR /code

# Install deps as root for cache-friendly layer; switch to user for runtime.
COPY --chown=user:user package.json ./
COPY --chown=user:user bun.lockb* ./
COPY --chown=user:user tsconfig.json ./

# `bun install --frozen-lockfile` if bun.lockb committed; falls back to a
# fresh install if not. Use `--production=false` so dev deps (drizzle-kit,
# typescript) are present for the developer LLM's typecheck step.
RUN bun install --frozen-lockfile 2>/dev/null || bun install

# Copy app source AFTER install for cache reuse on dep-only changes.
COPY --chown=user:user src/ ./src/
COPY --chown=user:user tests/ ./tests/

USER user

EXPOSE 3000

# Note: e2b.toml's start_cmd overrides this CMD at sandbox boot. CMD is
# kept for parity with `docker run` smoke tests outside e2b.
CMD ["bun", "run", "src/index.ts"]
```

# Build + push the atlas-hono-bun E2B template image.
# Usage: ./scripts/build-template.sh
# Requires: E2B_API_KEY env var set; npx + node 22+ available.

set -euo pipefail

cd "$SCRIPT_DIR/.."

echo "Building atlas-hono-bun E2B template (this takes ~2-4 min)..."
npx --yes @e2b/cli template create atlas-hono-bun

#!/usr/bin/env bash
# Build the template image locally (no push to E2B) and curl it.

set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
cd "$SCRIPT_DIR/.."

IMAGE_TAG="atlas-hono-bun:smoke"
CONTAINER_NAME="atlas-hono-bun-smoke"

echo "Building local image $IMAGE_TAG..."
docker build -t "$IMAGE_TAG" .

docker rm -f "$CONTAINER_NAME" 2>/dev/null || true

echo "Starting container on port 3000..."
docker run -d --name "$CONTAINER_NAME" -p 3000:3000 "$IMAGE_TAG"

echo "Waiting for Hono to come up..."
for i in {1..30}; do
  if curl -fsS http://localhost:3000/health > /dev/null; then
    echo "OK http://localhost:3000/health returns 200"
    curl -s http://localhost:3000/health
    echo ""
    echo "Cleanup: docker rm -f $CONTAINER_NAME"
    exit 0
  fi
  sleep 1
done

echo "FAIL: Hono did not come up within 30s. Logs:"
docker logs "$CONTAINER_NAME"
exit 1
```

# atlas-hono-bun E2B template

Bun 1.2+ runtime + Hono 4.x web framework + Drizzle ORM 0.36+ + Zod 3.x + postgres-js.

Used by Atlas's developer role when the user pins a project to this template via
